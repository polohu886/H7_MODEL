## 2. 迁移核心原则

### 2.1 先测试，后修改

- **第一步**：将驱动源码加入CMake编译，不改任何驱动代码
- **第二步**：在 `main.c` 中编写最简调用测试（单函数调用即可）
- **第三步**：编译烧录，观察运行结果
- **第四步**：只有测试失败才修改驱动代码

### 2.2 每次测试前清理旧测试代码

- 每次做新驱动测试前，**必须删除**上次在 `main.c` 中添加的测试代码
- 保持 `main.c` 整洁，只保留当前正在测试的驱动调用
- 迁移完成确认OK后，删除测试代码，恢复干净的while(1)循环
- 驱动初始化代码（如 `ZPN_UART_Init()`）可以保留在 `USER CODE 2` 区

### 2.3 不要提前优化

- 不要顺手重构驱动代码
- 不要添加"可能以后有用"的功能
- 只修必要的兼容问题

### 2.4 每次迁移成功后做 Git 提交

- 每完成一个驱动迁移（测试通过），**必须做一次 git commit**
- 提交信息格式：`迁移 MY_XXX 驱动至H743平台` 或 `Migrate MY_XXX driver to H743`
- 如果迁移过程中修改了其他文件（如 `CMakeLists.txt`、`stm32h7xx_hal_conf.h`、`stm32h7xx_it.c` 等），一并纳入同一次提交
- 好处：出问题时可以精确定位到是哪次迁移引入的，方便回滚

## 3. 迁移标准流程（SOP）

```
步骤1  →  将驱动 .c 文件加入 CMakeLists.txt 的 target_sources
步骤2  →  将驱动头文件路径加入 CMakeLists.txt 的 target_include_directories
步骤3  →  检查驱动所需的 HAL 模块是否已在 stm32h7xx_hal_conf.h 中使能
步骤4  →  检查驱动 #include 的 HAL 头文件名是否正确（F4→H7 差异）
步骤5  →  在 main.c 中编写最小测试代码
步骤6  →  编译 → 烧录 → 测试
步骤7  →  测试成功：删除测试代码，更新本文档迁移状态表
步骤8  →  **git commit**：提交所有变更，信息格式 `迁移 MY_XXX 驱动至H743平台`
步骤9  →  测试失败：排查（参考第4节常见问题），修改驱动代码，重新测试
```

## 4. STM32H7 关键差异（常见踩坑点）

### 4.1 内存管理 — 最重要！

**H7 有多块独立 SRAM，地址不同，DMA 可访问性不同：**

| 内存区域 | 地址 | 大小 | CPU访问 | DMA访问 |
|----------|------|------|---------|---------|
| DTCM | 0x20000000 | 128KB | 零等待 | **不可靠/不可达** |
| AXI SRAM | 0x24000000 | 512KB | 低延迟 | 正常 |
| D2 SRAM | 0x30000000 | 288KB | 稍慢 | 正常 |
| D3 SRAM | 0x38000000 | 64KB | 慢 | 正常 |

**当前链接脚本 (`STM32H743XX_FLASH.ld`) 已将 `.data` 和 `.bss` 放在 AXI SRAM (0x24000000)**，栈和堆仍在 DTCM。这是为DMA兼容做的修改。

**迁移检查清单：**
- 如果驱动使用了 DMA（UART DMA、ADC DMA、SPI DMA 等），DMA 缓冲区**不能**在 DTCM
- 静态变量/全局变量会自动放在 AXI SRAM（当前 `.bss`/`.data` 在 RAM 区），一般没问题
- 但如果驱动内部用 `__attribute__((section(".dtcm")))` 显式指定了 DTCM，需要改为 AXI SRAM 或 D2 SRAM
- 如果驱动内部用 `malloc` 从堆分配缓冲区，堆当前在 DTCM（`_sbrk` 从 `_end` 开始），DMA 访问堆内存会出问题
  - **解决方案**：关键DMA缓冲区用静态数组（走 `.bss` 在 AXI SRAM）或显式指定段

### 4.2 HAL 头文件名差异

F4 系列 → H7 系列，HAL 头文件名不同：

| F4 | H7 |
|----|----|
| `stm32f4xx_hal.h` | `stm32h7xx_hal.h` |
| `stm32f4xx_hal_uart.h` | `stm32h7xx_hal_uart.h` |
| `stm32f4xx_hal_gpio.h` | `stm32h7xx_hal_gpio.h` |

**驱动代码中如果有 `#include "stm32f4xx_hal_xxx.h"`，需要改为 `stm32h7xx_hal_xxx.h`**。

### 4.3 中断处理

- **NVIC 使能**：HAL 外设初始化后必须调用 `HAL_NVIC_EnableIRQ()`（UART/SPI/I2C等中断模式需要），有些 `HAL_xxx_MspInit()` 中可能漏掉
- **中断处理函数**：必须在 `Core/Src/stm32h7xx_it.c` 中添加对应的 `xxx_IRQHandler()`
- 中断函数名必须与启动文件 (`startup_stm32h743xx.s`) 中的向量表名称一致

### 4.4 HAL 模块使能

驱动所需的 HAL 外设模块必须在 `Core/Inc/stm32h7xx_hal_conf.h` 中取消注释（`#define HAL_xxx_MODULE_ENABLED`）。当前已使能的：
- ADC, DAC, DMA, MDMA, GPIO, I2C, TIM, UART, RCC, FLASH, EXTI, PWR, CORTEX, HSEM

如果驱动用到了 SPI、RNG、SAI 等，需要手动使能。

### 4.5 时钟配置

- 当前系统时钟：480 MHz (HSE 25MHz → PLL1)
- 外设挂在不同的总线（APB1/2/3/4），时钟频率不同
- 如果驱动依赖特定外设时钟频率，检查时钟树是否正确

### 4.6 MPU 配置

当前 MPU 配置了一个全局 No-Access 区域，部分子区域禁用。如果驱动需要使用特定的内存区域（如外部SDRAM、FMC等），可能需要调整 MPU 配置。

### 4.7 陶晶驰串口屏协议要点（HMI 驱动）

串口屏使用自定义指令协议，所有指令以 `\xFF\xFF\xFF`（3个0xFF）结尾。关键避坑点：

- **指令格式必须加 `.id` 后缀**：`cle s0,0xFF` → `cle s0.id,0xFF`，`add s0,0,100` → `add s0.id,0,100`。不加 `.id` 屏幕不识别
- **`addt` 快速波形指令时序**：
  1. 发指令头 `addt obj.id,ch,n` + `\xFF\xFF\xFF`
  2. **必须等 10ms**（`HAL_Delay(10)`）让屏幕解析指令并切换到数据接收模式
  3. 发 `n` 字节原始波形数据
  4. **必须发 `0x01`** 结束标记，否则屏幕永久卡在数据接收态
- **波形数据值域**：每个字节 0~255，屏幕映射到控件像素高度。控件 200px 高 → 数据 0~200 刚好满幅
- **TX/RX 中断竞争**：发送大批量数据时，屏幕可能回传数据触发 RXNE 中断，HAL 的 USART IRQHandler 优先处理 RX 再处理 TC，大量 RX 可能饥饿 TC 中断导致 DMA 回调链断裂。正常使用下短数据不受影响

## 5. CMakeLists.txt 修改模板

每次添加新驱动时，按以下格式修改 `CMakeLists.txt`：

```cmake
# 添加驱动源文件
target_sources(${CMAKE_PROJECT_NAME} PRIVATE
    # 已有驱动...
    MY_DRIVE/MY_NEW_DRIVER/driver.c
)

# 添加驱动头文件路径
target_include_directories(${CMAKE_PROJECT_NAME} PRIVATE
    # 已有路径...
    MY_DRIVE/MY_NEW_DRIVER
)
```

## 6. 测试不报错但功能失效的排查顺序

当驱动迁移完、编译通过、烧录运行无硬错误，但**功能不工作**时，按以下顺序排查：

1. **检查缓冲区地址**：打印或调试查看 DMA 缓冲区的地址是否在 DTCM（0x20000000-0x2001FFFF），如果是，需要移到 AXI SRAM
2. **检查 NVIC 中断是否使能**：在 `HAL_xxx_MspInit()` 中确认 `HAL_NVIC_EnableIRQ()` 被调用
3. **检查 IRQHandler 是否存在**：在 `stm32h7xx_it.c` 中确认中断处理函数存在
4. **检查 GPIO 时钟是否使能**：H7 的 GPIO 时钟由 `__HAL_RCC_GPIOx_CLK_ENABLE()` 使能，CubeMX 生成的代码通常在 `MX_GPIO_Init()` 中完成
5. **检查外设时钟是否使能**：确认 `HAL_xxx_MspInit()` 中有对应的 `__HAL_RCC_xxx_CLK_ENABLE()`
6. **检查 HAL 模块是否使能**：确认 `stm32h7xx_hal_conf.h` 中对应模块的 `#define` 未被注释

## 7. 已完成的修复记录（参考）

| 问题 | 原因 | 修复 |
|------|------|------|
| UART1 DMA TX 静默失败 | `.data`/`.bss` 在 DTCM，DMA 无法可靠访问 | 链接脚本将 `.data`/`.bss` 移到 AXI SRAM |
| UART2 中断不触发 | NVIC 未使能 + IRQHandler 缺失 | 补充 `HAL_NVIC_EnableIRQ(USART2_IRQn)` 和 `USART2_IRQHandler` |
| AD9959 输出频率 = 设置/20 | `Init_AD9959()` 从未被调用，PLL 未配置 | 在 main.c 添加 `Init_AD9959()` 调用 |
| Si5351 无输出 | `si5351_read8` 指针错误 `(uint8_t*)(&value)` → 写到指针自身；Integer Mode 设给了非 4/6/8 分频值 | 改为 `(uint8_t*)value`；仅 4/6/8 分频时使能 Integer Mode |
| FFT: 中断回调中计算死机 | `Phase_ComputeThdAndFft()` 在 DMA 中断上下文执行，浮点 DSP 运算导致 HardFault | 回调只做 memcpy+重启DMA，FFT 计算移到主循环 `FFT_SendSpectrumFrame` |
| FFT: snprintf %f 崩溃/输出空 | newlib-nano 下 `snprintf` 不支持 `%f` 浮点格式化 | 已修复: `-Wl,-u,_printf_float`启用浮点支持 + `_sbrk`修正堆边界(AXI SRAM用`_heap_end`替代DTCM的`_estack`) + 堆扩容至8KB |
| FFT: 采样率计算错误 | `Phase_Get_TIM3_TriggerFreq_Hz()` 用 APB1 时钟而未检测 Si5351 ETR 外部时钟 | 检测 `TIM_SMCR_ECE` 标志，ETR 模式用 1.024MHz；改 Period 后重新调用计算 |
| FFT: ADC 通道错误 | 未调用 `SCAN_ConfigAdc1ToFFTInput()`，沿用 CubeMX 默认 CH16 (PA0) | `FFT_App_Init()` 中调用配置为 PC4 (ADC_CHANNEL_4) |
| FFT: MATLAB 脚本过于复杂 | 原 574 行含谐波检测/波形分类/峰值标注，只需实时频谱 | 精简为 ~100 行纯频谱显示，加峰值频率标注 |
| DFT: 基频检测精度差 → 谐波 DFT 失谐 | 线性插值误差 ±62Hz，H3 频率偏差 3×。单频 DFT 带宽极窄，失谐导致幅值坍陷 | 改用抛物线插值精修，频率误差缩到 ±几Hz |
| DFT: DMA 边读边写数据碎片 | 无回调快照保护，`dft_adc_buf` 被 DMA 持续覆盖 | `HAL_ADC_ConvCpltCallback` 内 memcpy 到 `dft_snapshot`，DFT 只读 snapshot |
| DFT: 搜索范围硬编码 800~1200Hz | 只能测特定频率信号 | `DFT_App_Init` 增加 `search_min/max` 参数，默认全频段 1~256kHz |
| DFT: ADC 硬件配置散落在 main.c | 违反驱动封装原则 | `DFT_App_Init` 内完成 ADC 通道/TIM3/DMA 全部配置，main.c 只调接口 |
| HMI: `HMI_ClearWave` 无法清屏 | `cle s0,0xFF` 缺少 `.id` 后缀，屏幕不识别 | 改为 `cle s0.id,0xFF`，参考手册 `cle objname.id,ch` 格式 |
| HMI: `addt` 批量波形显示混乱/堵塞 | ①指令头与数据体间缺 `HAL_Delay(10)`，屏幕未切换模式 ②缺 `0x01` 结束标记，屏幕卡在数据接收态 | 保留延迟和结束标记。`addt` 协议要求：`\xFF\xFF\xFF` → 等待屏幕处理 → n字节数据 → `0x01` |
| HMI: `pow()` 引入 `math.h` 依赖 | 仅用于 `10^decimal` 查表，浮点开销大可避免 | 用 `static const int pow10[]` 替代 |
| HMI: USART1 函数交叉污染 `uart2_tx_busy` | USART1 的 KickLocked/OnTxComplete 错误操作了 UART2 忙标志 | 删除 UART1 函数中的 `uart2_tx_busy` 赋值（该标志仅 UART2 队列内部使用） |
| HMI: `HMI_FastWaveSend` 先入队再等空闲 | 超时后指令头残留 DMA 队列 | 空闲检查移到 `SendPrintf` 之前 |
